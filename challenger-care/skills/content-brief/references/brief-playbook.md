# Content Brief Playbook

## Full Brief Templates by Content Type

### Template 1 — Comprehensive Guide (Informational)

```
PAGE BRIEF: [Target keyword — e.g., "how to reduce customer churn"]
URL: /blog/[slug]
Target primary keyword: [Keyword]
Secondary keywords: [3–5 related terms from clustering]
Search intent: Informational
Target audience: [Role / knowledge level / goal — e.g., "SaaS founders and CS managers who understand churn is a problem but lack a systematic approach"]
Target word count: [1,800–2,800 words — based on SERP benchmark of top 3 results]
E-E-A-T tier: Medium (how-to content — first-hand experience preferred)

TITLE TAG: [Target keyword — Brand] (under 60 chars)
META DESCRIPTION: [155–160 chars — problem + outcome + CTA]
H1: [Closely matches title tag — same keyword]

INTRO (100–150 words):
- Open with the reader's pain point or context — NOT "In this article we will..."
- Establish credibility (first-hand experience reference or data point)
- State clearly what the reader will walk away with
[If Featured Snippet present: write a 40–60 word direct answer immediately after intro]

H2: [Core concept or definition — if needed for context]
  - Keep this short; readers who know the basics will skip it
  - Target: 150–200 words
  - Include: concise definition + why it matters for the reader specifically

H2: [First major strategy or step]
  - What to cover: [2–3 sentences of specific guidance]
  - Target: 300–400 words
  - Include: specific example, number, or outcome from practice
  H3: [Sub-tactic or nuance]
  H3: [Sub-tactic or nuance]

H2: [Second major strategy or step]
  - What to cover: ...
  - Include: ...
  H3: ...

H2: [Third major strategy or step]
  - ...

H2: Common mistakes to avoid [high E-E-A-T signal — shows first-hand knowledge]
  - 3–5 specific mistakes with why they fail and what to do instead
  - Target: 300–400 words

H2: Frequently Asked Questions
  - Source questions from People Also Ask for the target keyword
  - Each answer: 40–80 words, direct, structured for FAQPage schema
  - Target: 3–5 questions

CONCLUSION (100–150 words):
  - Summarise 3 key takeaways (not a repetition of the article)
  - CTA aligned to intent: if informational, direct to next learning step or tool

INTERNAL LINKS REQUIRED (outbound — this page links to):
  - Link to: [URL] using anchor text "[anchor]"
  - Link to: [URL] using anchor text "[anchor]"

INBOUND LINKS NEEDED (after publishing — editor adds these):
  - [Existing page URL] → add link using anchor "[anchor]"
  - [Existing page URL] → add link using anchor "[anchor]"

SCHEMA: Article or BlogPosting with datePublished, dateModified, author (name + title)
Add FAQPage schema to the FAQ section.

E-E-A-T NOTES:
  - Author should have [specific credential/experience]
  - Include one specific client example or first-hand data point in the first 300 words
  - Every statistic must link to its primary source (no secondary summaries)
  - Last-updated date must be visible on the published page
```

### Template 2 — Comparison Roundup (Commercial)

```
PAGE BRIEF: [Target keyword — e.g., "best project management tools for agencies"]
URL: /blog/[slug] OR /[slug] (if commercial nav item)
Target primary keyword: [Keyword]
Secondary keywords: [Tool names, niche variants, year-based variant]
Search intent: Commercial
Target audience: [Who — e.g., "agency operators comparing tools; likely decision makers; probably already using one tool and evaluating alternatives"]
Target word count: [2,200–3,000 words — match top 3 results]
E-E-A-T tier: Medium to High (requires testing/first-hand experience of each tool)

TITLE TAG: Best [Category] for [Audience] ([Year]) — Brand (under 60 chars)
META DESCRIPTION: [Problem + methodology + CTA — 155–160 chars]
H1: Best [Category] for [Audience] in [Year]

INTRO (100–150 words):
  - Why this comparison was needed (pain point)
  - How we evaluated: testing period, criteria, number of options reviewed
  - Quick summary: if reader is in a hurry, give the winner in one sentence

H2: Quick Comparison Table
  - Feature matrix: tools as rows OR columns, key criteria as the other axis
  - Include: pricing, best-for summary, free trial availability
  - Source from actual testing — not the tool's own marketing claims

H2: [Tool 1 Name] — Full Review
  H3: What [Tool 1] does well
  H3: Limitations (be honest — readers spot biased reviews immediately)
  H3: Pricing
  H3: Best for: [specific use case or team type]
  (Target: 300–400 words per tool)

H2: [Tool 2 Name] — Full Review
  [Same structure]

[Repeat for each tool — aim for 4–7 tools in most roundups]

H2: How to Choose — Buying Guide
  - Decision framework: if [condition], choose [tool]
  - 3–5 specific questions to ask before buying
  - Target: 300–400 words

H2: Frequently Asked Questions
  - Pull from PAA for the target keyword
  - 3–4 questions, 40–80 word answers
  - FAQPage schema applied

CONCLUSION (100–150 words):
  - Clear recommendation by use case (not "it depends" — be specific)
  - CTA: free trial link, comparison guide download, or consultation

INTERNAL LINKS REQUIRED:
  [Specify exact pages and anchors]

SCHEMA: Article + BreadcrumbList + FAQPage (FAQ section)

E-E-A-T NOTES:
  - Must include actual testing data (screenshots, real numbers, dates)
  - State the methodology explicitly in the intro: how long tested, what tasks performed
  - Update annually minimum — stale roundups lose rankings when tools update pricing/features
  - Author bio must mention direct experience with these tool categories
```

### Template 3 — Product / Service Landing Page (Transactional)

```
PAGE BRIEF: [Target keyword — e.g., "CRO agency for Shopify"]
URL: /[slug] (not /blog/ — this is a landing page)
Target primary keyword: [Keyword]
Secondary keywords: [Location variants, niche variants, service variants]
Search intent: Transactional
Target audience: [Who — e.g., "Shopify store owners doing $500K–$5M, frustrated with stagnant conversion rates, ready to hire a specialist"]
Target word count: 800–1,400 words (clarity converts; length does not)
E-E-A-T tier: High (service/purchase decision — trust signals critical)

TITLE TAG: [Service] — [Key Benefit] | Brand (under 60 chars)
META DESCRIPTION: [Outcome + differentiator + CTA — 155–160 chars]
H1: [Service Name] — [Key Outcome for the Client]

HERO SECTION:
  - Value proposition: what we do, for whom, and the outcome they get — in 1–2 sentences
  - Primary CTA (above the fold): [e.g., "Book a Free Audit"] — prominent, not buried

H2: What's Included / What You Get
  - Benefit-led: "You get X that does Y" not "We offer Z feature"
  - Specific outcomes: numbers, timelines, deliverables
  - Target: 200–300 words

H2: How It Works
  - 3–4 steps: numbered process shows clarity and builds confidence
  - Target: 200–250 words

H2: [Specific Use Case or Client Type]
  - Show how the service solves a specific version of the problem
  - Include one client example or outcome (with permission)
  - Target: 150–200 words

H2: Results / Social Proof
  - Testimonials: name, company, specific outcome (not generic praise)
  - Case study link if available
  - Data points: "Average client sees X improvement in Y time"

H2: Pricing / How to Get Started
  - Be transparent — hiding pricing increases bounce rate from the wrong visitors
  - Clear next step: what happens when someone clicks the CTA?

H2: Frequently Asked Questions
  - Address the top 3–5 objections as FAQ format
  - FAQPage schema applied
  - Target: 3–5 questions

FINAL CTA:
  - Repeat the primary CTA
  - Add urgency or social proof element ("Join 40+ brands we've helped")

INTERNAL LINKS REQUIRED:
  [Specify]

SCHEMA: Service (or LocalBusiness if location-specific) + FAQPage + BreadcrumbList
Consider AggregateRating if you have enough reviews

E-E-A-T NOTES:
  - Client logos or recognisable brand names (with permission) dramatically increase trust
  - Named testimonials with outcomes outperform anonymous ones 3:1
  - Publication date AND last-updated date visible
  - Clear contact information and team page link
```

### Template 4 — SaaS Competitor Comparison (BOFU)

```
PAGE BRIEF: [Target keyword — e.g., "Klaviyo vs Salesforce"]
URL: /[competitor]-vs-[brand] OR /[brand]-vs-[competitor]
Target primary keyword: [Keyword]
Secondary keywords: ["[competitor] alternative", "[competitor] vs [brand] [year]", "[brand] vs [competitor] pricing"]
Search intent: Commercial (BOFU — user is in active evaluation)
Target audience: [Who — e.g., "mid-market ops managers currently using or trialling Klaviyo, evaluating Salesforce as they scale; likely have a budget, likely have a deadline"]
Target word count: 1,800–2,500 words
E-E-A-T tier: Medium to High (comparative claims must be accurate and current)

TITLE TAG: [Competitor] vs [Brand]: [Key differentiator] (under 60 chars)
META DESCRIPTION: [155–160 chars — who each is best for + link to comparison]
H1: [Competitor] vs [Brand]: Which Is Right for You in [Year]?

INTRO (100–150 words):
  - Honest framing: "Both are strong tools. This helps you decide which fits your situation."
  - State the key differentiator in one sentence (teaser for the conclusion)
  - Who this guide is written for

H2: Quick Summary (for skimmers)
  - 1-paragraph plain-English recommendation: "[Tool A] is better for X; [Tool B] is better for Y"
  - Comparison table: key features, pricing, best for — 6–8 rows

H2: [Competitor] — Strengths and Best Use Cases
  - Be genuinely accurate: readers know the competitor's product; dishonest coverage = lost trust
  - 3–4 specific strengths with examples
  - Who gets the most value from it
  - Target: 300–400 words

H2: [Brand] — Strengths and Best Use Cases
  - Same structure as above
  - Where Brand wins vs competitor (this is the sales argument, but make it factual)
  - Target: 300–400 words

H2: Feature-by-Feature Comparison
  - Table format: feature, how [Competitor] handles it, how [Brand] handles it, winner
  - Include: onboarding, integrations, support, pricing model, scalability
  - Note where competitor wins honestly — it builds trust in your Brand wins

H2: Pricing Comparison
  - Be specific: actual prices, not "contact us for pricing" cop-outs
  - Total cost of ownership if relevant: implementation, training, add-ons
  - Target: 200–300 words

H2: What [Competitor] Customers Say When They Switch [if applicable]
  - Real review language from G2/Capterra about pain points with the competitor
  - Connects to your solution without being manipulative
  - Target: 150–200 words

H2: Who Should Choose [Competitor]
  - Be honest: acknowledge who the competitor genuinely serves better
  - Paradox of honesty: admitting this builds trust in your [Brand] recommendation

H2: Who Should Choose [Brand]
  - Clear, specific criteria: "If you [condition], [Brand] is the better choice"

H2: FAQs
  - Pull from PAA: "Is [Competitor] better than [Brand]?", "[Competitor] vs [Brand] pricing", etc.
  - FAQPage schema applied

FINAL CTA: Free trial / demo — direct, one action

INTERNAL LINKS REQUIRED:
  [Specify — link to own pricing page, features page, case studies]

SCHEMA: Article + FAQPage + BreadcrumbList
If review data is embedded: AggregateRating on the [Brand] section

E-E-A-T NOTES:
  - Verify all competitor pricing and features before publishing — inaccuracies = credibility damage
  - Set a reminder to review and update this page every 6 months (pricing and features change)
  - Author should have direct experience with both tools or have spoken with users of both
```

## SERP Feature → Content Format Decision Table

| SERP feature observed | Format instruction for the brief |
|----------------------|-----------------------------------|
| Featured Snippet (paragraph) | Write a 40–60 word direct answer to the query question, placed in the first substantive section after the intro. Use clean HTML (`<p>` tag, no surrounding clutter). |
| Featured Snippet (numbered list) | Structure the core answer as an ordered list with 4–8 items. Each item should be a complete, brief sentence. |
| Featured Snippet (bulleted list) | Structure as an unordered list. Items should be noun phrases or short parallel sentences. |
| Featured Snippet (table) | Include a comparison table answering the query directly. At least 3 columns, 4+ rows. |
| People Also Ask (PAA) | Each PAA question = potential H2 or H3. Write a 40–80 word answer to each. Apply FAQPage schema. |
| AI Overview present | Write the most direct, citable answer to the primary query in the first substantive section. Use numbered steps or short paragraphs. Apply relevant schema. |
| Shopping results | Do not write a blog post. The intent is transactional. Build a product or category page instead. |
| Video carousel | Consider whether a companion video would help. At minimum, embed one relevant video from YouTube if one exists. |
| Image Pack | Include 3+ high-quality images with descriptive alt text. Original images (not stock) signal E-E-A-T. |
| News/Top Stories | For timely topics: add a "last updated" date prominently. For evergreen topics: avoid competing with news — focus on comprehensive, timeless content. |

## E-E-A-T Signal Checklist by Tier

### Tier 1 — High E-E-A-T (financial, medical, legal, safety, civic topics)

- [ ] Author has verifiable, relevant credentials (e.g., CPA for financial content, MD for medical)
- [ ] Author credentials displayed in byline AND author bio with credential details
- [ ] Every factual claim cited with a link to a primary source (study, official documentation, data)
- [ ] Medical/legal disclaimer present where required by topic
- [ ] Publication date AND last-reviewed date visible
- [ ] Expert peer reviewer credited if applicable
- [ ] Contact information for the organisation visible on page or in footer
- [ ] No unsourced claims, no "some say", no vague attributions

### Tier 2 — Medium E-E-A-T (product reviews, comparisons, how-to guides, educational content)

- [ ] First-hand experience referenced explicitly (e.g., "We tested this for 30 days", "When we ran this with a client...")
- [ ] At least one specific example or case study with real outcomes (not hypothetical)
- [ ] Key statistics cited with source links
- [ ] Author attributed with relevant background (even a one-line bio helps)
- [ ] Publication date visible; last-updated date visible if content is older than 6 months
- [ ] Methodology or approach stated explicitly (how did you arrive at this? what criteria did you use?)

### Tier 3 — Standard (general informational, brand storytelling, awareness content)

- [ ] Author attributed (not "Staff writer" or anonymous)
- [ ] Information is accurate and up-to-date as of publishing date
- [ ] External links to primary sources for statistics and data
- [ ] Publication date visible

## Schema Implementation Reference

| Content type | Schema type | Required fields | Optional but valuable |
|-------------|------------|----------------|----------------------|
| Blog post / guide | `Article` or `BlogPosting` | `headline`, `datePublished`, `dateModified`, `author` (with `name`), `publisher` | `image`, `wordCount`, `description` |
| Comparison page | `Article` + `BreadcrumbList` | Same as Article + breadcrumb hierarchy | `AggregateRating` if reviewing a specific product |
| Product page (eCommerce) | `Product` | `name`, `offers` (with `price`, `priceCurrency`, `availability`) | `aggregateRating`, `review`, `brand`, `image`, `sku` |
| Service landing page | `Service` | `name`, `provider`, `description` | `areaServed`, `hasOfferCatalog`, `aggregateRating` |
| FAQ section | `FAQPage` | `mainEntity` array of `Question`/`Answer` pairs | — |
| How-to guide | `HowTo` | `name`, `step` array with `HowToStep` | `totalTime`, `image` for each step |
| SaaS / Software page | `SoftwareApplication` | `name`, `applicationCategory`, `operatingSystem` | `aggregateRating`, `offers` |
| Local business | `LocalBusiness` | `name`, `address`, `telephone` | `openingHours`, `geo`, `sameAs` |
| Author bio | `Person` | `name` | `jobTitle`, `url` (author page), `sameAs` (LinkedIn/Twitter) |
| Organisation | `Organization` | `name`, `url`, `logo` | `sameAs` (social profiles), `contactPoint` |

**Implementation note:** HowTo and FAQPage were deprecated for rich results in 2024–2025. Do NOT stop implementing them — AI systems (Gemini, ChatGPT, Perplexity) cite pages with structured schema data 4× more than pages without it. The rich result benefit is gone; the AI citation benefit is growing.

**JSON-LD placement:** Always implement schema as JSON-LD in the `<head>` or at the bottom of `<body>`, not as Microdata in HTML attributes. JSON-LD is easier to maintain and less error-prone than Microdata.

**Validation:** Before publishing, test every schema implementation using Google's Rich Results Test (https://search.google.com/test/rich-results). Zero validation errors required.
